## B) 棚板プレビューの判定フロー（箇条書き）

1. **入口**  
   - `main.js` の `canvas.addEventListener("mousemove", …, true)` で、キャプチャ phase の mousemove を単一登録している。
2. **板モードでないとき**  
   - `window.isBoardModeOn` が false なら、板プレビュー用の分岐には入らない。
3. **板モードのとき**  
   - `hidePreview` / `hideLegPreview` / `resetHighlight` を実行。  
   - `updateRayFromClientXY(e.clientX, e.clientY)` で `mouse` を更新し、`raycaster.setFromCamera(mouse, camera)` でレイを更新。
4. **ワールド座標の取得**  
   - `raycaster.intersectObjects([...beams.map(b=>b.mesh), ...poles.map(p=>p.mesh)], false)` で hit を取得。  
   - **`hit.length > 0`** のときだけ、**`hit[0].point`** を `worldPos` として使う。  
   - `hit.length === 0` のときは、地面平面 `y=0` とレイの交点を計算し、交差すればその点を `worldPos`、交差しなければ `hideBoardPreview()` のみ実行して終了。
5. **プレビュー表示の可否**  
   - `showBoardPreview(worldPos)` を呼ぶ。  
   - その中で `findNearbyJoints4(worldPos, 1500)` により「worldPos に最も近い beam/pole から同じ面のジョイントを広げ、4本で囲まれた四角形」を探す。  
   - `j4 == null` なら `hideBoardPreview()` して終了。  
   - 同じ j4 のときはメッシュ更新をスキップし、それ以外なら一度 `hideBoardPreview()` したうえで新しいプレビューメッシュを生成・表示する。

要約すると、「プレビューが表示される」のは  
「mousemove → Raycaster が beam/pole メッシュに当たる → `hit[0].point` を `worldPos` にできる → `findNearbyJoints4(worldPos)` が j4 を返す」  
という経路が成立する場合に限られる。

---

## C) 狭くなっている一番の原因（1つに絞る）

**Raycaster の交差対象が「beam/pole の細いメッシュ表面だけ」になっていること。**

- プレビュー用の `worldPos` は、**`hit.length > 0` のときだけ** `hit[0].point` から得ている。  
  つまり、「カーソル認識」できる範囲 = Raycaster が beam/pole メッシュと交差する範囲、に完全に依存している。
- beam/pole は `createCylinderBetween` で作られ、`CylinderGeometry(4, 4, baseLen, 16)`（半径 4）の円柱である。  
  シーン単位が mm なら、半径 4mm の細い棒であり、画面上の投影は線に近く、数ピクセル幅程度しかない。
- `raycaster.params` の threshold などは未設定のため、交差は「レイがそのメッシュのポリゴンに当たるかどうか」の純粋な幾何判定であり、認識領域を広げる処理はどこにもない。
- その結果、マウスがわずかにずれただけで `hit.length` が 0 になり、プレビューが消えたり、地面フォールバックで意図しないセル／なしになって「出たり消えたり」する。

したがって、**「認識領域が狭い」最大の原因は、レイが当たる対象が「半径 4 の円柱メッシュ表面のみ」であり、当たり判定が細かすぎること**と結論できる。

---

## D) 解決策を3案（A: invisible hit mesh / B: threshold 拡大 / C: hysteresis）

### 案A: Invisible Hit Mesh（不可視の当たり用メッシュ）

- **概要**  
  各「セル」（4本の beam/pole で囲まれた矩形）に対し、見えない薄い平面メッシュ（または厚みの小さい Box）を用意し、`intersectObjects` の対象に「beam/pole の mesh」に加えてそれらを入れる。  
  レイは主にその平面に当たり、`hit[0].point` で広い範囲から安定して `worldPos` が取れる。
- **変更箇所**  
  - セル単位で hit 用メッシュを生成・保持する処理（例: `js/parts/boards.js` か、板専用のモジュール）。  
  - ラック構築／変更時に「有効なセル一覧」を求め、各セルに一枚の PlaneGeometry/Box を置く。  
  - `main.js` の板モード時の `intersectObjects` の配列に、その hit 用メッシュのリストを追加する（`recursive` は現状どおりで可）。  
  - レイに当てるだけなので、`visible = false` または `material.visible = false` で描画から外す。
- **副作用**  
  - セル候補の事前計算・更新が必要になり、ジョイント／ビーム／ポールの増減・移動のたびに hit メッシュを再構築する必要がある。  
  - オブジェクト数が増え、`intersectObjects` の対象が増えるが、メッシュはシンプルなので負荷はあまり大きくない想定。  
  - 既存の「beam/pole に直接当てる」挙動は、hit リストの並びによっては hit[0] が hit メッシュになるため、板モード時は「hit メッシュ優先」とみなして `hit[0].point` を使う設計にする必要がある。
- **最小差分の見積もり**  
  - セル列挙＋hit メッシュ生成ロジックを 1 か所にまとめ、`main.js` の板モード分岐で intersect 対象を「beam/pole のみ」から「beam/pole + セル hit メッシュ」に変えるだけにすれば、変更は「板モード周り＋セル計算の出しどころ」に集中する。  
  既存の `findNearbyJoints4` はそのまま `worldPos` を受け取るので、入口側の変更は小さい。

---

### 案B: Threshold 拡大（レイの「太さ」で許容範囲を広げる）

- **概要**  
  Three.js の `Raycaster` の `params.threshold` は主に Line/LineSegments 用であり、通常の Mesh の交差には使われない。  
  そのため、**Mesh に対する「当たりの許容範囲」を広げる**には、実装側で「交差しなくても、レイに近いオブジェクトを『当たった』とみなす」処理が必要になる。  
  つまり「threshold 拡大」＝「レイから一定距離以内の beam/pole を、あたかも hit したように扱い、その線分上の最近点などを `worldPos` にする」方式を指す。
- **変更箇所**  
  - `main.js` の板モード分岐内、`intersectObjects` の後。  
    `hit.length === 0` の場合に、全 beam/pole について「レイと線分（ジョイント間）の最短距離」を計算し、その距離がしきい値（例: 50mm）以内なら、その線分上の最近点を `worldPos` として `showBoardPreview(worldPos)` を呼ぶ。  
  - 必要なら、`findNearbyJoints4` で使う `worldPos` の意味が「厳密な hit 点」から「レイに近い線上の点」に変わることだけ、コメントや変数名で明確にする。
- **副作用**  
  - 「狙っていない beam/pole」がしきい値内に入ると、意図しないセルがプレビューされる可能性がある。  
  - 複数ある場合は「レイに最も近い線分」を選ぶなど、優先順位を決める必要がある。  
  - しきい値を単位（mm）で持つため、カメラ距離や解像度に応じて同じしきい値で「狭い／広い」と感じる違いは残る。
- **最小差分の見積もり**  
  - `main.js` の板モード分岐だけを修正し、`intersectObjects` の結果に加えて「レイ–線分の距離 < 閾値」のフォールバックを一枚噛ませる形にすれば、他ファイルを触らずに済む。  
  `findNearbyJoints4` や `showBoardPreview` のシグネチャはそのままでよい。

---

### 案C: Hysteresis（履歴で表示を安定化）

- **概要**  
  「一度セルが有効になったら、マウスが少し外れてもしばらく同じプレビューを表示し続ける」ようにする。  
  レイが beam/pole に当たらなくなった直後は即 `hideBoardPreview` せず、  
  - 直近で有効だった `worldPos` や `j4` を保持し、  
  - 「レイが何にも当たらない状態が N ms 続く」「または別のセルに明確に移った」まで、前のプレビューを維持する。
- **変更箇所**  
  - 板モード用に「最後に有効だった worldPos / j4 / 時刻」を保持する変数（例: `main.js` の bindMouseHandlers のスコープ内、または boards.js のモジュール変数）。  
  - `main.js` の mousemove 板モード分岐で、  
    - `hit.length > 0` のときは現行どおり `worldPos = hit[0].point` とし、その値と時刻を保存して `showBoardPreview(worldPos)`。  
    - `hit.length === 0` のときは、即 `hideBoardPreview()` せず、「前回の有効から一定時間以内かつ、別の j4 がまだ確定していない」なら、前回の `worldPos` で `showBoardPreview` を呼び続ける（または前回の j4 でプレビューを再表示する）。  
  - 時間で切る場合は `setTimeout` や `requestAnimationFrame` で「無効が続いたら hide」するか、mousemove のたびに「最後に有効だった時刻」と「最後に無効になった時刻」を比較して判定する。
- **副作用**  
  - マウスをセルから離したあと、短時間だけプレビューが残るため、「もう外れた」という感覚とずれる可能性がある。  
  - 閾値時間・距離のチューニングが必要。  
  - 「狭い」こと自体は解消されないが、「出たり消えたり」はかなり抑えられる。
- **最小差分の見積もり**  
  - 板モードの分岐内と、必要なら `boards.js` の `showBoardPreview` 周りに、状態を保持する変数と分岐を足すだけになる。  
  `intersectObjects` の対象や `findNearbyJoints4` のロジックは変えずに済む。

---

## まとめ

- **原因**  
  プレビュー用の `worldPos` が「Raycaster が beam/pole メッシュに当たったときの `hit[0].point`」に強く依存しており、当たり対象が半径 4 の細い円柱だけなため、認識領域が画面上で数ピクセル幅程度しかなく、そこで「狭い」「出たり消えたり」が起きている。
- **対応の方向性**  
  - **根本的に広げる** → 案A（invisible hit mesh）でセル面を当たりとして追加する。  
  - **当たり判定を緩める** → 案B（threshold 相当の「レイに近い線分」フォールバック）で、交差しなくても近い beam/pole から `worldPos` を取る。  
  - **見た目の安定だけ取る** → 案C（hysteresis）で、一度出たプレビューを短時間維持する。

本調査時点ではコードの修正は行っておらず、上記はすべて「提案」とする。
